# Game Build Prompt: Tic-Tac-Toe vs. Bot

## Mission
You are a game developer. Your job is to build, thoroughly test, and check in a complete,
playable game called "Tic-Tac-Toe vs. Bot" to a git repository. Follow every section of
this prompt exactly. When you finish, the repo must contain working game code and a passing
test suite.

---

## Game Overview
Tic-Tac-Toe vs. Bot is a single-player browser-based game where a human plays against a
computer opponent on a 3×3 grid. Before each game, the player chooses whether to play as X
or O — the bot takes the other symbol. X always goes first; if the player chose O, the bot
moves first automatically. The first to place three marks in a row (horizontally, vertically,
or diagonally) wins. If all nine squares are filled with no winner, the game is a draw.
The bot has two difficulty modes: Hard (plays optimally via minimax — cannot lose) and Easy
(makes a random legal move — can be beaten).

## Functional Requirements

Implement every item below. Each is numbered for traceability.

1. **Symbol selection screen** — Before showing the board, display a screen with the game
   title and two buttons: "Play as X" and "Play as O". No other UI is shown until the player
   clicks one.

2. **Symbol assignment confirmation** — After the player selects a symbol, display a
   persistent label during gameplay indicating both assignments, e.g. "You: X  |  Bot: O".

3. **Difficulty selection** — On the same symbol selection screen (or immediately after
   symbol choice), the player selects difficulty: "Easy" or "Hard". Default to Hard if not
   yet chosen. This selection persists until changed.

4. **3×3 game board** — Render a 3×3 grid of nine equal clickable cells, all initially empty.

5. **Player move** — When it is the player's turn, clicking an empty cell places the player's
   symbol in that cell and passes the turn to the bot.

6. **Invalid move prevention** — Clicking an occupied cell, or clicking any cell while it is
   the bot's turn, must have no effect (no error, no state change).

7. **Bot turn — Hard mode** — The bot uses the minimax algorithm to select the optimal move.
   It will never lose; the best outcome for the player is a draw.

8. **Bot turn — Easy mode** — The bot selects a random empty cell. It makes no attempt to
   block or win strategically, so a skilled player can reliably beat it.

9. **Bot move delay** — The bot always waits 400ms before placing its mark, regardless of
   difficulty, so the move does not appear instantaneous.

10. **Automatic bot-first move** — If the player chose O, the bot (playing X) takes its first
    move automatically as soon as the board is displayed, after the standard 400ms delay.

11. **Win detection** — After every move, check all eight winning lines:
    - Rows: [0,1,2], [3,4,5], [6,7,8]
    - Columns: [0,3,6], [1,4,7], [2,5,8]
    - Diagonals: [0,4,8], [2,4,6]
    If any line is completely filled by one player, that player wins.

12. **Draw detection** — If all nine cells are filled and no winning line exists, the game
    ends in a draw.

13. **Winning line highlight** — When a win is detected, visually distinguish the three
    winning cells from the rest of the board (e.g., different background color or bold border).

14. **Result message** — When the game ends, display one of:
    - "You win!" (player wins)
    - "Bot wins!" (bot wins)
    - "It's a draw!" (draw)
    The board must become non-interactive after the result is shown.

15. **Play Again button** — After a result is displayed, show a "Play Again" button. Clicking
    it clears the board and returns to the symbol selection screen. All game state is reset.

## Game Rules and Logic

Keep all game logic in `game.js` as pure functions with no DOM access. This makes them
independently testable.

### Board representation
The board is an array of 9 values: `cells[0..8]`, each `null | 'X' | 'O'`.
Index layout:
```
0 | 1 | 2
---------
3 | 4 | 5
---------
6 | 7 | 8
```

### Turn order
- X always moves first.
- If the player is X: player moves first, then bot.
- If the player is O: bot (X) moves first, then player.
- Turns strictly alternate. The bot never moves twice in a row.

### Win condition
A player wins if they occupy all three cells in any of the eight winning lines listed in
requirement 11. Check after every move (both player and bot).

### Draw condition
If `cells` contains no `null` values and no winner has been found, the game is a draw.

### Minimax (Hard mode)
Implement a standard minimax function:
- `minimax(board, isMaximizing, botSymbol, playerSymbol)` → returns the best score
- Bot maximizes; player minimizes
- Terminal states: bot wins (+10), player wins (-10), draw (0)
- No alpha-beta pruning required, but may be added for clarity
- The bot always picks the move with the highest minimax score

### Easy mode
Select a random index from the list of empty cells (`cells.filter(c => c === null)`).

## Input Handling

- **Click / tap on empty cell (player's turn):** Place player's mark. Pass turn to bot.
- **Click / tap on occupied cell:** No effect.
- **Click / tap on any cell (bot's turn):** No effect.
- **Click "Play as X" / "Play as O":** Set player symbol, set difficulty if shown, show board.
- **Click "Easy" / "Hard" toggle:** Update difficulty setting immediately.
- **Click "Play Again":** Reset all state, return to symbol selection screen.
- Touch events must work identically to mouse click events (standard browser behavior via
  `click` event on touch screens is acceptable).

## Testing Requirements

Split tests across two files: `tests/game.test.js` (unit) and `tests/integration.test.js`.

### Unit Tests — `tests/game.test.js`

Test every exported function from `game.js`. All tests must be pure (no DOM, no timers).

Required test cases (write these at minimum; add more as needed):

**checkWinner(cells)**
- Returns the winning symbol when a row is complete (test all 3 rows)
- Returns the winning symbol when a column is complete (test all 3 columns)
- Returns the winning symbol when a diagonal is complete (test both diagonals)
- Returns null when the board is empty
- Returns null when the board is full with no winner (draw state)
- Returns the correct winning line indices alongside the winner

**isDraw(cells)**
- Returns true when all 9 cells are filled and there is no winner
- Returns false when the board has empty cells
- Returns false when there is a winner (even if all cells are filled — winner takes precedence)

**getEmptyCells(cells)**
- Returns all 9 indices when the board is empty
- Returns correct indices after some cells are filled
- Returns an empty array when the board is full

**getBotMoveEasy(cells)**
- Returns an index that is currently empty
- Never returns the index of an occupied cell
- Returns a value between 0 and 8 inclusive

**getBotMoveHard(cells, botSymbol, playerSymbol)**
- Picks the winning move when the bot can win in one move
- Blocks the player's winning move when the player can win in one move
- Returns center (index 4) on an empty board (optimal opening)
- Never returns the index of an occupied cell
- Returns a valid index (0–8) in all cases

**minimax(board, isMaximizing, botSymbol, playerSymbol)**
- Returns +10 (or positive) when bot has won
- Returns -10 (or negative) when player has won
- Returns 0 for a drawn board

### Integration Tests — `tests/integration.test.js`

These tests simulate full game sequences using `game.js` logic only (no DOM). Wire up moves
manually by setting `cells` and calling game functions in sequence.

Required scenarios:

1. **Player (X) wins via top row** — Simulate moves: player takes 0, bot takes 4, player takes
   1, bot takes 8, player takes 2. Assert `checkWinner` returns X and correct line indices.

2. **Bot (Hard, X) wins when given the opportunity** — Set up a board where bot has two in a
   row and one empty cell to complete it. Assert `getBotMoveHard` returns the winning cell.

3. **Bot (Hard) blocks player from winning** — Set up a board where the player has two in a
   row and one empty cell to complete it, and it is the bot's turn. Assert `getBotMoveHard`
   returns the blocking cell.

4. **Full game ends in draw with two optimal players** — Simulate a complete game where both
   player and bot always use `getBotMoveHard`. Assert the game ends in a draw (no winner,
   all cells filled).

5. **Player chooses O — bot moves first** — Confirm that on a fresh board with player as O,
   `getBotMoveHard` returns a valid move (bot, playing X, goes first).

6. **Easy bot loses to a perfect player** — Simulate a game where the player uses minimax and
   the bot uses `getBotMoveEasy`. Assert that it is possible for the player to win (the easy
   bot does not always draw or win).
